
# 🚇 서울시 지하철 대중교통 데이터 레이크 수집 정의서

본 문서는 서울 열린데이터 광장(data.seoul.go.kr)의 공공데이터를 활용하여, **[서울시 지하철역 일별 이용 트래픽과 교통약자 편의시설 간의 상관관계 분석 마트]** 를 구축하기 위한 데이터 정의 및 엔지니어링 수집 전략을 기록합니다.

## 🛠️ 1. 기술 스택 & 인프라 (Tech Stack)
- Language: Python 3.14+
- Orchestrator: Apache Airflow 3.2 (LocalExecutor , StandAlone)
- Database: PostgreSQL 18 (OLAP 및 Raw Data 저장소) Ubuntu 26.04 
- Environment: Window 10
---

## 🎯 기획 의도 및 데이터 선정 사유

단순한 데이터 수집(Dump)을 넘어, **"일별 이용객 수가 가장 혼잡한 지하철역들에 교통약자를 위한 이동 편의시설(엘리베이터 등)이 충분히 확보되어 있는가?"** 라는 비즈니스 가설을 검증하고자 합니다. 이를 위해 데이터의 성격에 맞춰 수집 데이터셋을 선정했습니다.

---

## 🏗️ 데이터 모델링 및 소스 수집 전략 (Fact & Dimension)

본 파이프라인은 엔지니어링 최적화를 위해 데이터를 매일 쌓이는 **실적(Fact)** 데이터와, 자주 변하지 않는 **기준 정보(Dimension)** 데이터로 이원화하여 수집을 진행합니다.

### 📊 A. 비즈니스 실적 데이터 (Fact Table)
* **대상 데이터**: 서울시 지하철 호선별 역별 승하차 인원 정보 (`CardSubwayStatsNew`)
* **선정 사유**: 일별/역별 이용 트래픽의 변화 추이와 혼잡도를 추적하여 분석 마트의 핵심 계량 지표(Measure)로 활용하기 위함입니다.
* **수집 방식**: **Open API (JSON 포맷 전환 연동)** 
* **파이프라인 이유**: 해당 데이터는 평일/주말 관계없이 매일 새벽에 전일자 확정 데이터가 자동으로 누적 업데이트됩니다. 사람이 매일 손으로 다운로드할 수 없으므로, **Apache Airflow 스케줄러를 활용한 무인 데일리 배치 자동화 파이프라인**을 구축하기 위해 API 연동 방식을 채택합니다.

### 🗺️ B. 마스터 프로필 데이터 (Dimension Table)
* **대상 데이터**: 
  1. 서울시 역사 마스터 정보 (`subwayStationMaster` - 위경도 및 주소)
  2. 서울시 지하철역 엘리베이터 위치정보 (`SubwayElevator` - 편의시설 수량)
* **선정 사유**: 실적 팩트 테이블에 존재하는 단순 '역 이름'에 위치 좌표와 엘리베이터 보유 수량이라는 구체적인 프로필(속성)을 결합하여 분석 차원을 확장하기 위함입니다.
* **수집 방식**: **웹 포털 CSV 파일 벌크 로드 (Initial Bulk Load)**
* **파이프라인 이유**: 역사 위치나 엘리베이터 수량은 새 역이 개통하거나 시설 증설이 없는 한 데이터가 거의 변하지 않는 준정적(Quasi-Static) 특성을 가집니다. 이를 매일 API로 중복 호출하는 것은 자원 낭비이므로, **최초 1회 CSV 파일로 전체 데이터를 벌크 적재(Full Refresh)한 후 월간 단위로 동기화**하는 아키텍처가 시스템 최적화에 가장 부합하여 이 방식을 채택합니다.

## 📋 수집 대상 데이터셋 파라미터 명세

| 데이터 분류 | API/데이터셋 한글명 | 시스템 식별자 | 스케줄러 주기 | 수집 포맷 |
| :--- | :--- | :--- | :--- | :--- |
| **Fact (실적)** | 서울시 지하철 호선별 역별 승하차 인원 정보 | `CardSubwayStatsNew` | 매일 새벽 (`@daily`) | API (JSON) |
