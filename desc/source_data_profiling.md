# 🕵️‍♂️ 서울시 지하철 원천 데이터 프로파일링 및 API 규격 역공학 보고서

본 문서는 서울 열린데이터 광장의 지하철 일별/시간별 승하차 원천 데이터(`subwSttn`) XML 응답 결과를 직접 분석하여, 데이터의 제약 조건, 스키마 구조, 그리고 데이터 파이프라인(EL) 설계 방향성을 역추적(Reverse Engineering)한 기록입니다.

---

## 📊 1. 원천 데이터 분석을 통해 발견한 핵심 인사이트 (Data Insights)

1. **대용량 데이터 볼륨 확인 (`<list_total_count>`)**
   - 원천 서버 내부의 총 레코드 수가 **398,680건**으로 식별되었습니다.
   - 오픈 API의 1회 호출 제한(1,000건)을 고려할 때, 전체 역사적 데이터(Historical Load)를 유실 없이 적재하려면 최소 400회 이상의 분할 요청(Pagination Loop) 전략이 필수적입니다.

2. **데이터 적재 입도(Grain)의 재정의 (`<GTNF_SE>`)**
   - 기존 예상과 달리, '승차'와 '하차' 실적이 한 레코드의 컬럼으로 분리되어 있지 않고, **동일한 날짜/역임에도 승차 행(Row)과 하차 행(Row)이 별도의 레코드로 분리되어 적재**되어 있음을 발견했습니다.
   - 따라서 데이터베이스 ODS 테이블 설계 시 고유키(Primary Key) 조합에 반드시 `승하차구분(gtnf_se)` 컬럼이 포함되어야 멱등성(Idempotency)이 보장됩니다.

## 📐 2. 원천 데이터 필드 명세 및 추론 데이터 타입 (Schema Mapping)

분석한 XML 구조를 바탕으로 PostgreSQL 18 데이터베이스에 생성할 ODS 테이블의 물리적 데이터 타입을 아래와 같이 추론 및 정의합니다. (PostgreSQL 표준에 의거하여 전 컬럼 소문자 변환 원칙 적용)

| XML 태그명 | 추론 데이터 타입 | 컬럼 설명 | 데이터 샘플 및 제약 조건 |
| :--- | :--- | :--- | :--- |
| `sn` | **INTEGER** | 일련번호 | `1`, `2` (내부 시퀀스 번호) |
| `mvmn_ymd` | **VARCHAR(10)** | 이동년월일 | `2024-01-01` (YYYY-MM-DD 포맷 문자열) |
| `line` | **VARCHAR(50)** | 지하철 호선명 | `1호선`, `경의중앙선` 등 |
| `sttn_no` | **INTEGER** | 고유 역사 번호 | `150` (역사 마스터와 조인 가능한 Key) |
| `sttn` | **VARCHAR(100)** | 지하철 역명 | `서울시`, `시청`, `종각` |
| `gtnf_se` | **VARCHAR(10)** | 승하차 구분 | `승차` 또는 `하차` (핵심 분기 지표) |
| `hr06_bfr` ~ `hr24` | **INTEGER** | 시간대별 이용객 수 | `383`, `257` 등 (음수가 없는 순수 정수형태) |

## 🚀 3. 파이프라인(Airflow 3.2) 설계 반영 사항

- **URL 호출 포맷 강제 변환**: 원천 데이터가 XML로 제공되나, 파이썬 판다스(Pandas) 라이브러리와의 호환성 및 메모리 효율성을 위해 API 호출 시 URL 패스를 `json`으로 강제 전환하여 수집 프로세스를 최적화합니다.
- **오류 방어 전략**: 시간대별 필드(`hr06` 등)에 간혹 공백이나 결측치(Null)가 들어올 가능성에 대비하여, 데이터프레임 적재 시 `fillna(0)` 처리를 거친 후 DB에 `Bulk Insert` 하도록 코드를 방어적으로 구성합니다.
